**Best approach:** In the `teamPts` CTE, don't group by yet -- keep individual matchup rows. Then in the `standings` CTE, do the grouping:

The cleanest path: Keep the existing `teamPts` CTE as-is (it returns aggregated gamePts and xp per team). Add a new `matchCounts` CTE that counts matchup weeks per team from the schedule+matchResults join:

```sql
matchCounts AS (
  SELECT teamID, COUNT(*) AS matchWeeks FROM (
    SELECT sch.team1ID AS teamID
    FROM matchResults mr JOIN schedule sch ON mr.scheduleID = sch.scheduleID
    JOIN currentSeason cs ON sch.seasonID = cs.seasonID
    UNION ALL
    SELECT sch.team2ID AS teamID
    FROM matchResults mr JOIN schedule sch ON mr.scheduleID = sch.scheduleID
    JOIN currentSeason cs ON sch.seasonID = cs.seasonID
  ) x GROUP BY teamID
)
```

Then in `standings` CTE, join `matchCounts` and compute: `(mc.matchWeeks * 3) - SUM(tp.gamePts) / 2 AS losses` (since wins = gamePts/2, and total games per week = 3, losses = totalGames - wins, where totalGames = matchWeeks * 3). But wait -- ties: each tied game gives 1 pt to each team (not 2 or 0). So `gamePts / 2` doesn't cleanly equal wins when ties exist. If a game ties, each team gets 1 game point, so gamePts counts the tie game as 0.5 in the "wins" formula.

Actually in the current code: `wins = gamePts / 2` -- this gives fractional wins (e.g. 4.5) because ties give 1 pt. The display shows this as "4.5W". So losses = `matchWeeks * 3 - wins` where wins includes half-wins from ties. This means losses also has halves. Example: 3 games, 1 win (2pts), 1 tie (1pt), 1 loss (0pts) = gamePts 3, wins = 1.5, losses = 3 - 1.5 = 1.5. That's correct: 1W, 1T, 1L maps to 1.5W, 1.5L.

So: `losses = (matchWeeks * 3) - wins`. Add this computation. The interface gets `losses: number` (decimal, like wins).
