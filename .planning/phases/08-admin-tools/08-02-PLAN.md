  <action>
1. Create `src/lib/admin/lp-api.ts` -- refactor LP API calls from `scripts/import-week-scores.mjs`:
   - `lpPullScores(cookie: string, seasonID: number, weekNum: number): Promise<{matches: StagedMatch[], warnings: string[]}>` -- fetches scores from LP API, maps team names to DB teamIDs (use the LP_TEAM_MAP from the script), fuzzy-matches bowler names to DB bowlers using Fuse.js, returns structured StagedMatch array with unmatched bowlers flagged
   - Preserve the LP_LEAGUE_ID and LP_BASE constants from the script
   - Use the existing bowler matching logic from import-week-scores.mjs (query all bowlers from DB, fuzzy match with Fuse.js, flag unmatched)
   - Return clear error on LP 401/403 ("LP session expired, please get a fresh cookie")

2. Create `src/lib/admin/validation.ts` -- score validation (ADMN-02):
   - `validateScores(matches: StagedMatch[]): ValidationWarning[]`
   - Checks per research: score > 280 = warning, score < 50 = warning, score deviates > 80 pins from avg = info, duplicate bowlerID in same match = warning
   - Skip validation for penalty rows (isPenalty)
   - Return array of warnings (never blocking, just displayed)

3. Create `src/lib/admin/scores.ts` -- refactored pipeline logic from `scripts/import-week-scores.mjs`, `scripts/populate-match-results.mjs`, `scripts/populate-patches.mjs`:
   - `insertScores(seasonID: number, week: number, matches: StagedMatch[]): Promise<{inserted: number, personalBests: PersonalBest[]}>` -- INSERT into scores table (only: bowlerID, seasonID, teamID, week, game1, game2, game3, incomingAvg, turkeys, isPenalty). NEVER insert computed columns (hcpGame1/2/3, handSeries, incomingHcp, scratchSeries). Calculate incomingAvg per bowler before insert using the same logic as import-week-scores.mjs. After insert, query for any new personal bests (high game, high series) and return them.
   - `runMatchResults(seasonID: number, week?: number): Promise<number>` -- refactored from populate-match-results.mjs. Delete existing match results for the week first, then regenerate from schedule + scores.
   - `runPatches(seasonID?: number): Promise<string[]>` -- refactored from populate-patches.mjs. Return list of newly awarded patches.
   - `deleteScoresForWeek(seasonID: number, week: number): Promise<number>` -- delete existing scores for re-entry
   - `bumpCacheAndPublish(seasonID: number, week: number): Promise<void>` -- bump `.data-versions.json` scores channel, clear relevant cache files. Does NOT publish (that's a separate step).
   - ALL DB operations use `withRetry()` from db.ts. The confirm pipeline runs SEQUENTIALLY: delete old scores -> insert -> match results -> patches -> bump cache. No parallelization (Azure SQL 30-connection limit).

4. Create API routes:
   - `src/app/api/admin/scores/pull/route.ts` POST: accepts `{ cookie: string, seasonID: number, week: number }`, calls `lpPullScores()`, returns `{ matches: StagedMatch[], warnings: string[] }`. Requires admin auth.
   - `src/app/api/admin/scores/confirm/route.ts` POST: accepts `{ seasonID: number, week: number, matches: StagedMatch[] }`, runs the full pipeline (delete old -> insert -> match results -> patches -> bump cache), returns `{ inserted: number, personalBests: PersonalBest[], patches: string[] }`. Requires admin auth.
   - `src/app/api/admin/scores/manual/route.ts` POST: accepts `{ seasonID: number, week: number }`, returns empty StagedMatch array scaffolded from schedule data (teams matched up for that week) so admin can manually fill in scores. Requires admin auth.

Create `src/app/admin/scores/page.tsx` as a client component ('use client') implementing the full score review workflow:

**Layout:**
- Top bar: Season selector (dropdown of seasons), Week selector (dropdown 1-20+), "Pull from LP" button, "Manual Entry" button
- Season/week default to current season and latest week

**Step 1 - Pull/Enter:**
- "Pull from LP" opens a modal/panel for pasting the LP connect.sid cookie, then calls POST /api/admin/scores/pull
- "Manual Entry" calls POST /api/admin/scores/manual to get empty match scaffolds
- Loading state while pulling
- Error display for LP failures (clear message about expired cookie)

**Step 2 - Review (main UI):**
- Display matches as cards: each card shows "Team A vs Team B" header
- Within each card, a table of bowlers with columns: Name, Game 1, Game 2, Game 3, Turkeys, Avg, Status
- All score fields are editable inputs (number type)
- Turkey count field inline per bowler (not a separate step per CONTEXT.md)
- Unmatched bowlers highlighted in yellow with a "Resolve" button that opens fuzzy search dropdown (use Fuse.js against bowler list fetched from API) + "Create New Bowler" option
- Penalty/absence toggle per bowler row (sets isPenalty, fills 199 per game per HCP formula)
- Validation warnings displayed as non-blocking banners below each match card (yellow for warnings, blue for info)
- Run validation client-side using the same validateScores logic (import from shared lib or inline)
- "Add Bowler" button per team within a match card for manual additions

**Step 3 - Confirm:**
- "Confirm & Process" button at bottom (disabled until at least one match has scores)
- Calls POST /api/admin/scores/confirm with the full staged data
- Loading overlay during processing ("Inserting scores... Running match results... Calculating patches...")
- On success: show summary panel with personal bests hit, patches awarded, number of scores inserted
- "Back to Scores" button to start another week

**Editing previous weeks:**
- Changing season/week dropdowns and pulling/entering allows editing any week
- A warning banner if scores already exist for selected week: "Scores exist for this week. Confirming will replace them."

**Styling:**
- Tailwind with project palette (cream background, navy headers, red accents)
- Match cards with subtle shadows, clear visual grouping
- Responsive: stacks vertically on mobile, horizontal scroll for wide tables
